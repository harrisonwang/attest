# attest — 立项实证：可行性探针记录

> 日期：2026-07-09 · 脚本：[probe/drift-probe.py](probe/drift-probe.py)（原样归档，含已知缺陷注记）
> 结论先行：**绑定式抽取在真实文件上成立。** 5 份真实 agent 文档、约 230 个 token：绑定率 60–90%，按"只报 BROKEN、其余沉默"策略零误报；每个覆盖缺口都映射到一个可写的机械 resolver，无一映射到"需要 NLU"。

## 1. 背景

立项讨论中的核心质疑：*"CLAUDE.md/AGENTS.md/SKILL.md 没有强格式规范，无法抽取就无法核验。"*

探针要回答的问题因此不是"能不能理解文档"，而是：**不理解文档、只做 CommonMark 提取 + 仓库绑定，在真实文件上能拿到多高的绑定率、多低的误报率？** 120 行一次性 Python，约 90 分钟，无任何 LLM 参与。

## 2. 方法

1. 提取：inline code span 全取；fenced 块（bash/sh/空标注）逐行取命令。
2. 绑定：每个 token 依序尝试——路径存在性（含 basename 近失搜索）→ package.json scripts / Makefile targets → workspace 包名 / `--filter` 目标 → PATH 二进制 → ALL_CAPS 环境变量 grep → 标识符符号 grep。
3. 裁决：绑上 = 计数；形态完备但绑不上 = BROKEN；其余 = 沉默（不报）。

## 4. 六个教训 → 设计决策

每条教训在 [01-design.md](01-design.md) 中的落点：

| # | 现象 | 归因 | 设计落点 |
|---|------|------|---------|
| 1 | `cargo test`、`git commit` 等行内多词命令未绑定 | 探针只对 fenced 行做命令解析 | 行内 span 与 fenced 行走同一命令行解析器（§3） |
| 2 | SKILL.md 的 `phases/` 报 BROKEN，实际相对 SKILL.md 自身存在（5 个文件俱在） | 只按仓库根解析 | 四级基目录序，doc-dir 最优先（§6） |
| 3 | ASCII 流程大纲混入 token（"Phase 1: Discovery…"） | 无语言标注的 fence 被当 shell | 只解析显式 bash 标注的 fence（§3） |
| 4 | `net/http`、`mattn/go-sqlite3` 报 BROKEN-path | Go import 路径被路由到 path resolver | go-import resolver，对 go.mod + stdlib 表绑定（§5#5） |
| 5 | `ask_audience` 报 BROKEN-symbol，实为 `.scaffold-docs.yml` 里真实存在的键 | 无配置键命名空间 | config-key resolver（§5#7） |
| 6 | jadeenvoy 的 `CONVENTIONS.md` 报 BROKEN，但原文是"未配置；可建 symlink"——文档自己声明它不存在 | 无语境判断 | finding 携带所在行原文 + 否定语境守卫，broken 降 suspect（§7）；严格语境判定归银档 |

另有正向发现：`/v1/runtimes`、`POST /v1/sessions/:id/events`（API 路由）与 `agt-`、`sess-`（ID 前缀字面量）目前沉默，但都可被机械 resolver 覆盖（路由注册表 grep / 字符串字面量 grep）——记入 P4 生态清单。

## 5. 探针证明了什么、没证明什么

**证明了**：

- 反引号约定在真实 agent 文档中足够普遍，CommonMark 提取即可拿到高密度可绑定 token；
- "仓库为 oracle"的绑定判定精度极高——所有假 BROKEN 均来自探针实现缺陷（缺 resolver / 缺基目录规则），而非方法本身；
- 误差不对称成立：未绑定的都安静地沉默了，没有一例"理解错句子"式误报。

**没证明**（留给 P0 语料实验）：

- 金档对**历史真实 drift** 的检出率（探针测的是当前快照的绑定率，不是对已知 drift 的召回）——这正是 P0 挖矿语料要测的，kill criteria 60% 悬在这里；
- 反引号外散文指涉的占比与形状候选的实际精度；
- 大型多语言 monorepo 上的性能与作用域规则表现。
