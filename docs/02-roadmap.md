# attest — 分阶段路线

> 状态：立项方案 · 2026-07-09
> 原则：每个阶段有明确的交付物和止损条件——数据不支持就停或者改方向，不带感情续命。
> 时间提醒：agents-lint（见 00-vision 的竞品分析）2026 年 3 月已经发了 v0.5，窗口按月算。前三个阶段加起来控制在八周以内。

## 第一阶段：验证想法（第 1 到 2 周）

先证明这件事值得做，再写产品代码。

写一个挖矿脚本，扫 git 历史，找出"commit A 改了代码，后面 commit B 修了文档"的配对。这种配对就是带天然标注的真实 drift 案例——包含文档位置、错误的 token、修正后的 token、间隔了多长时间。扫的范围是自己的仓库（spoor、vouch、answer-trace 等几个）加上 15 到 20 个知名开源仓库，优先挑有 CLAUDE.md 或者 AGENTS.md 的。目标是攒 200 个以上的案例。

对每个案例做人工分类，看它落在路径、脚本、命令、符号、环境变量、配置项、散文这七类里的哪一类。得出真实的频率分布，知道哪些检查模块最值钱。同时把之前那版探针脚本按已知缺陷修一轮——加上行内命令解析、文档自身目录作为基目录、只认有 bash 标注的围栏块——然后在案例上测覆盖率和误报。

交付物是 `corpus/` 案例库入库，加一份数据报告（分布、覆盖率、误报清单）。

止损条件是两条。如果金档检查模块对历史真实 drift 案例的覆盖率不到 60%，说明散文占比超出了预期，架构得往银档方向前置。如果探针修复版在案例上仍然有无法归因到"缺检查模块"的系统性误报，说明铁律第一条不成立，方案作废。

## 第二阶段：做出能用的版本（第 3 到 5 周）

搭 attest-core 和 attest-cli 的骨架，Rust 写的，跟 vouch 同一个技术栈。先把提取器做出来——用 pulldown-cmark 解析 Markdown，加上命令行解析。然后实现前六个检查模块：路径、脚本、包名、命令、环境变量、符号。Go import 和配置项这两个看第一阶段的数据决定是否进 MVP。

加上裁决逻辑、语境守卫、基线机制、还有 `--since` 增量模式。报告输出支持三种格式：终端默认的 TTY、`--format json`（机器可读的 attest.report.v1 协议）、`--format github`（GitHub PR 批注）。测试全写 golden——每一条规格都有对应的测试用例，语料回归跑进 CI，attest 自己检查自己的文档。

交付就是 `attest check` 在自己全部仓库上能跑，语料回归的误报为零。

验收标准是 MVP 的检出率不低于探针修复版。拿五个从没见过的开源仓库做冷启动运行，人工复核所有 broken，不能有一条误报。

止损条件是如果为了压误报不得不把主力检查模块关到覆盖率掉下 40%，那回到第一阶段重新看数据。

## 第三阶段：分发和发布（第 6 到 8 周）

建好四个分发渠道。npm 上发 `@harrisonwang/attest`（包了二进制，`npx` 就能用，零安装）。GitHub Action 直接用仓库里的 `action.yml`。Claude Code 里配上 `/attest` skill 和 SessionStart hook。brew 和 scoop 复用 spoor 的 tap 和 bucket。

扫 top-50 开源仓库，从六万个 AGENTS.md 采用者里挑高星的。产出一份结构化的发现报告——哪个仓库的文档对 agent 撒了什么谎。挑其中确认度高的、修起来直白的 drift，给上游提 PR。只发确认度高的，每个 PR 都带着工具名曝光，一个误报都不能有。

launch 的内容线是：我们扫了 50 个知名仓库的 agent 文档，其中 X% 在对 agent 撒谎——附上方法和工具。HN、X、中文社区各发一轮。

交付是可安装的 v0.1、launch 文章、至少 10 个上游 PR。没有硬性的验收门槛，但如果 launch 后三十天内没有任何有机使用信号，后续降速为兴趣项目节奏。

## 第四阶段：lock 和银档（第 9 到 12 周，第三阶段有信号后启动）

实现 `.attest/claims.lock`——先做纯机械档，形状候选加绑定确认。再做 `attest extract`，用 LLM 辅助从散文里抽取断言，跟现有项目的 OpenAI 配置一样用。锚点必须绑定成功才落盘，proposed 走 git review 到 approved。加上哈希锚点复查，锚点代码变了就发 suspect 提醒。

跟 vouch 的 IR 打通——IR 里声明的变更面直接定向触发相关绑定的复查，是增量模式的精确版。配上 `/attest-fix` skill，agent 读 report 改文档复跑到绿，这就是"文档自维护"的演示视频，第二波传播素材。

## 第五阶段之后：按真实需求牵引

符号搜索从 grep 升级到 tree-sitter，TS、Python、Rust、Go 都支持，只会更准不会翻案。Go import、路由注册、配置项这些检查模块继续深化。文档面扩展到 README、docs 站点、mdbook 和 docusaurus 的源文件。加一个显式的 opt-in，允许跑仓库自产二进制的 `--help` 来获取更准的命令信息。做 MCP server 给沙箱化的 agent 场景用，VS Code 诊断插件也考虑。工具表社区化，数据文件开 PR 通道。

## 不碰的事情

不生成文档——那是 scaffold-docs 的事，attest 只当裁判。不做散文风格检查、不打文档结构分——那是 AgentLinter 们的地盘，而且太主观。不查链接——lychee 已经做得很好了，最多集成。不做通用测试框架。CI 里绝对不调 LLM——第三条铁律是产品的公开承诺，写死在 README 第一屏。API 面守住，只留 check、baseline、extract 三个子命令。

## 怎么算做成了

精度上，语料回归的误报永远为零——这是每次发布的硬门禁。覆盖上，挖矿语料的 broken 检出率目标 55% 以上，broken 加 suspect 总检出目标 80% 以上。采纳上，launch 后 90 天内有明确的有机使用信号就行。声誉上，上游 PR 合并率不低于 70%——低于这个数说明确认度纪律出了问题。
