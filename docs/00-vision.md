# attest — 为什么做这件事

> 状态：立项方案 · 2026-07-09

## 问题在哪

文档写出来就开始过时。目录改名了，脚本重命名了，命令行参数删了，API 变了——代码在往前走，文档不知道。没人能做到写完代码回头一条条检查文档还有没有效，这件事软件行业几十年没解决过，大家只是忍着。

以前忍忍也就算了。但现在 AI 编程工具重度依赖 CLAUDE.md、AGENTS.md、SKILL.md 这些文档，读它们的不是人，是每回都逐字照做的 agent。你文档里写了个早就删掉的脚本，agent 跑一次撞一次墙，白白烧 token，还带着错误的预设继续往下干活。文档从以前那种"有空翻翻的参考资料"变成了要命的运行时依赖，而运行时依赖理应有测试。

外部信号也印证了这不是杞人忧天。

AGENTS.md 已经被 Anthropic、OpenAI、Block 一起捐进了 Linux Foundation 下面的 Agentic AI Foundation（2025 年底）。六万多个项目在用。标准化意味着更大的存量，更大的存量意味着更快的腐化速度。2026 年已经有研究测出来：过时或低质量的上下文文件让任务成功率掉了 2% 到 3%，成本涨了 20% 以上。同类工具也陆续冒头，后面会聊到，说明不止一个人看到了这个痛点。

## 我们的判断

attest 建立在一个更根本的判断上：AI 负责生成，机器负责公证。

模型帮你想、帮你写、帮你总结，但"它说的还对不对"这件事不能靠另一个模型来验——那是用一个幻觉检查另一个幻觉。验证必须是确定性的、可复现的、跟模型无关的基础设施。

这个判断不是凭空来的，在三个姊妹项目里已经验证过三轮了。answer-trace 公证的是 LLM 回答里的事实能不能在源文档里逐字找到。vouch 公证的是 commit 声明跟实际代码改动是否吻合。git-why 公证的是 AI 对改动的解释符不符合代码。attest 公证的是文档对仓库的引用还做不做数。

模型越强、产出越多，"它说的是真的吗"这个问题总量就越大。生成类工具可能被下一版模型吃掉，但验证类工具只会随着模型变强被更需要。这是独立开发者少有的、不跟巨头抢饭碗的方向。

## 凭什么能行

直觉上这条路不好走：文档没有格式规范，你怎么把里面有用的断言抽出来？怎么知道作者那句"去跑 src/test.sh"是在说真实路径还是随口举例？

attest 的答案是把问题反过来想——不问作者什么意思，只问仓库认不认。

第一步稳得很。Markdown 有标准的解析规则，反引号（`code`）和围栏代码块（```bash）的语法是明确无误的。开发者写文档有个近乎普遍的习惯——可执行的东西、可引用的文件全都用反引号包着，因为本来就是写给读者照着敲的。你说"去读一下 `Cargo.toml`"，那个 token 就是给 attest 的明确信号。

第二步是关键。拿到 token 之后，attest 不猜它代表什么。它换个问：这个 token 能在文件树里找到吗？在 package.json 的 scripts 里能查到吗？是 workspace 的包名吗？能在源码里搜到这个符号吗？仓库本身就是裁判。哪个角度能给一个肯定的回答，token 就归哪类。

这条路径能走通的根本原因是不对称性。attest 漏掉一条真断言的代价是少了一点覆盖率，但永远不会因为理解错了句子而把对的说成错的。核验的单位不是"这句话什么意思"，而是"这东西还能不能对上"。覆盖率可以慢慢从 60% 爬到 90%，精度从第一天就是硬的。CI 工具死于误报，不死于漏报。

2026 年 7 月 9 号我们拿 120 行一次性脚本在 5 份真实的文档上做了摸底。大概 230 个 token，绑定率 60% 到 90%，零误报——唯一边界案例自己指出了修复方式。每一个没覆盖到的缺口都对到一个能机械实现的检查逻辑，不需要任何语义理解。详细数据和分析在 `03-probe-2026-07-09.md` 里。

## 竞品怎么看

最接近的是 [agents-lint](https://github.com/giacomo/agents-lint)。口号几乎一样——"Your AGENTS.md is probably lying"。目前是 TypeScript 写的，npm only，检查路径存在、npm scripts、依赖版本，以及一些跟具体框架绑定的模式（React、Laravel 之类），有交互式的 fix 功能。

它的存在是好事——说明痛点是真实且被多方独立发现的，不是我们在意淫。但时机也紧，窗口可能就几个月。

差异化不在功能的多少，在架构的根本不同。agents-lint 是一组规则清单：路径规则写一套，npm 规则写一套，框架 pattern 再写一套。attest 是一个判定引擎：统一从 token 出发，尝试多个角度绑定，绑定状态直接映射到裁决。往引擎里加一个新生态只需要写一个新的检查模块，不是写一整套新规则。

其他维度的差距也很明显。agents-lint 只覆盖 npm，attest 一开始就覆盖了 npm、cargo、go、python、make 这些多语言生态。精度纪律上更不一样——attest 的铁律是拿不准就闭嘴，agents-lint 有些启发式判断（靠年份引用、框架时代特征之类）误报空间更大。attest 还多了基线机制，老项目第一天就能接进来不用先修存量；多了 claims.lock，能追踪"上次还在这次没了"这种退化；多了机器可读的报告协议，agent 可以自动修完再跑一遍确认变绿。

[Swimm](https://swimm.io) 是另一个维度的产品，走企业级路线，要你写文档的时候用他们的编辑器把片段挂到代码行上，靠 git 历史推测同步。attest 立场相反——零作者负担，不要求任何人改变写文档的习惯，事后从任意 Markdown 里抽。两者面向不同场景，可以长期共存。

还有一些是互补关系而非竞争。AgentLint 和 AgentLinter 做的是文档结构打分和 LLM 评估，不做仓库事实核验。cclint 和 claudelint 只校验 Claude Code 的配置文件格式。lychee 专门查死链，很成熟，将来可以集成进来。doctest 类的工具只执行代码块，不管引用关系。oasdiff 盯着 API spec 和实现的偏差，赛道不同但未来可能有交叉。

一句话概括：agents-lint 是一组规则，attest 是一个判定引擎。规则清单的天花板是作者想到的检查，绑定引擎的天花板是验证模块的积累——而积累就是门槛。

## 为什么不容易被抄

用之前 spoor 项目总结的五条标准来检验。

attest 卡在一个必经的位置上——每次 CI 跑、每个 agent session 启动，都要面对"文档还对吗"这个问题。它的正确性是客观可测的，一个绑定要么成立要么不成立，golden corpus 可以回归验证。它有长尾积累效应——每种语言的 import 语法、路由框架的约定、配置文件的格式、CLI 工具的子命令表，以及从 git 历史里挖出来的真实 drift 案例，这些都随时间复利增长。代码可以抄，语料和经验抄不走。它的接口面很小，核心就一个动词 check，外加 baseline 和 extract。它在 CI 里是确定的、本地的、便宜的——零 LLM 调用、毫秒级、结果可复现。

摸底实验已经演示了积累的形态。一次运行暴露的每个盲点——Go import 被当成路径、YAML 键找不到绑定对象、相对基目录设错了——都精确对到一个 20 行的确定性检查逻辑，不需要"提升模型理解力"。三年后这个检查模块库就是一条护城河，跟 spoor 积累的 ZIP 炸弹防御一样——时间换来的、别人追不上的资产。

## 命名的事

2026 年 7 月 9 号查了一下各个平台的命名占用。

crates.io 上 `attest` 已经被占用了（一个叫"Dead simple test framework for the age of AI"的项目，2026 年 6 月还在更新），但 `attest-core` 和 `attest-cli` 可以用。npm 上的裸名 `attest` 也被一个九年前的 a11y 库占了。另外 GitHub 自己有个 Artifact Attestations 功能（`actions/attest`），在供应链安全语境下占了"attest"的搜索流量。

决定不变。attest 这个词意思太准了——出庭作证，文档的每一句话都要能在仓库面前站得住。跟 spoor、sift、vouch、glyph 这几个单音节动词也一脉相承。发布名跟 spoor 一样走带前缀的路线：crate 叫 `attest-core` 和 `attest-cli`，npm 叫 `@harrisonwang/attest`（走 npx 零安装），GitHub Action 叫 `harrisonwang/attest-action`，brew 和 scoop 复用 spoor 的 tap 和 bucket。搜索策略上不跟供应链安全语境正面抢，关键词打"attest docs"和"文档一致性检查"。

## 风险的事

AI 编程工具的开发商可能在自家产品里内置类似功能。如果真的发生，文档验证这个切入点可能会被收编。但引擎收编不了——多生态、多语言、通用文档的判定能力超出了任何单一工具的边界。换个滩头继续（比如 README、docs 站点），资产不丢。

agents-lint 可能靠先发优势滚雪球。但它 npm only、规则式的架构决定了重写成绑定引擎约等于重做。窗口以月算，不以年算，前两个阶段必须快。

误报毁声誉是最大的风险。解决办法是把精度纪律写死进设计公理。对外发修复 PR 只用确认过的项目。基线机制保证老项目第一天接进来不会炸。

覆盖率可能不及预期，大量文档引用绑不上。这就是第一阶段挖矿实验的任务——golden corpus 对历史 drift 案例的覆盖率如果不到 60%，就说明方向有问题，得重新设计。

名字撞车前面说过了，接受现状绕行。
